## Context

一個微型戰鬥原型。目標不是做出完整遊戲，而是用最小可玩迴圈把一個 GAS 技能的戰鬥手感做到位。開發於 UE5 5.x。

## Goals / Non-Goals

**Goals**
- 一個能玩 30 秒的核心戰鬥迴圈：移動 → 瞄準 → 放火球 → 消耗魔力 → 等冷卻 → 打假人。
- 至少一個技能透過 GAS 完整跑通（Ability + Attribute + Effect），證明「對 GAS 有概念」。
- 戰鬥有反饋（受擊、血條、魔力條），讓 demo「有手感」而非打死物。
- 文件層面規劃彈反機制作為下一步。

**Non-Goals（明確不做，用以守住範圍）**
- 連線 / 多人（超出原型範圍）。
- 程序化關卡、Roguelite 的道具/隨機（用文件描述視野即可）。
- 敵人 AI、尋路、攻擊行為。假人只被動受擊。
- 存檔、選單、結算畫面。
- 美術產出。全部用 UE5 內建範本素材與基本材質。
- **彈反機制本次不實作**，只寫進 GDD 作為「下一步」。

## Decisions

### 用 GAS 實作火球，而非單純 Blueprint 計時器
GAS 是 UE 連線技能戰鬥的官方標準框架。即使本原型不連線，用 GAS 做火球能以「正規做法」呈現魔力/冷卻（Attribute + GameplayEffect），並讓日後擴充（連線、更多技能）有一致基礎。
- 替代方案：純 Blueprint 用變數與 Timer 管理魔力/冷卻——更快，但無法展示 GAS 概念，捨棄。

### AttributeSet 用最小 C++，其餘 Blueprint
GAS 的 AttributeSet 在純 Blueprint 下體驗不佳，官方慣例是用 C++ 定義屬性（Health、Mana）。因此只寫一個最小 C++ AttributeSet，Ability/Effect/UI 都留在 Blueprint，兼顧「展示 C++ 能碰」與「開發速度」。
- 替代方案：全 Blueprint GAS——可行但繞路且不符官方慣例，捨棄。

### 以 UE5 Third Person 範本為基礎
移動與攝影機不是原型重點，直接沿用範本省下數天，把時間投在 GAS 與戰鬥反饋。
- 替代方案：從零搭角色——浪費時間在非核心項，捨棄。

### 火球為「投射物命中」而非「即時射線」
投射物能製造飛行時間與命中反饋的空間，戰鬥手感較好，也為未來「彈反」（假人發射投射物、玩家彈回）鋪路——兩者共用投射物基礎。
- 替代方案：Hitscan 射線——更簡單但手感平、且無法延伸彈反，捨棄。

### 彈反只寫文件、不實作（本次）
彈反是動作 roguelite 中最能展現設計理解的機制，但完整做（時機窗口、彈回、判定）會超出原型範圍。策略是：在 GDD 把設計講清楚（時機窗、成功/失敗反饋、與投射物系統的關係），先聚焦把火球的核心迴圈做完整。

## Implementation Contract

**可觀察行為（demo 跑起來應看到的）**
- 啟動 PIE，玩家以第三人稱角色出生於訓練室，可用 WASD 移動、滑鼠轉視角。
- 按滑鼠左鍵：若魔力足夠且不在冷卻中，角色發射一顆火球投射物，魔力條下降，技能進入冷卻（冷卻中再按無效）。
- 火球命中訓練假人：假人血條下降、播放受擊反饋（閃白或擊退其一即可）。
- 假人血歸零：短暫失能後在原位重置滿血。
- HUD 常駐顯示玩家魔力條與假人血條，數值隨戰鬥即時更新。

**資料形狀（GAS）**
- AttributeSet（C++）：`Health`、`MaxHealth`、`Mana`、`MaxMana`。
- Fireball GameplayAbility（BP）：CostEffect 扣 Mana、CooldownEffect 定義冷卻、命中時對目標套用 DamageEffect 扣 Health。

**驗收標準**
- 冷卻與魔力任一不足時，技能不觸發（不可繞過）。
- 假人受擊與死亡重置流程可重複觸發不崩潰。
- HUD 數值與實際 Attribute 一致。

**範圍邊界**
- 只有一個技能、一個假人、一個房間。不含彈反、AI、連線、存檔、美術。

## Risks / Trade-offs

- [GAS 學習曲線陡，可能吃掉多數時間] → 嚴守單一技能範圍；卡關時先用純 BP 版火球確保「有可玩 demo」，再回頭換 GAS。
- [C++ AttributeSet 對新手是門檻] → 跟隨官方/社群教學的標準樣板，屬性只留 Health/Mana 四個，不自創架構。
- [範圍蔓延（想加敵人/關卡）] → Non-Goals 白紙黑字；任何新想法一律寫進 GDD 的「下一步」而非本次實作。

## Migration Plan

不適用（全新 demo，無部署/回滾）。交付即 UE5 專案可 PIE 執行 + `docs/GDD.md`。

## Open Questions

- 受擊反饋用「閃白材質」還是「擊退位移」？→ 擇一即可，實作時選較快者。
- HUD 用 UMG 進度條 binding 還是事件更新？→ 先用 binding，夠用即可。
