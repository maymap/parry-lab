# Parry Lab — Game Design Document

> 一份求職作品集用的微型戰鬥原型。開發者為 4+ 年後端/全端工程師，UE5 與遊戲開發為新領域，本專案為邊學邊做的成果。
> **定位：低完成度原型（Prototype）。目標不是完整遊戲，而是用最小可玩迴圈證明「能動手用 GAS 做出有手感的戰鬥」，並展示對動作戰鬥設計的理解。**

---

## 1. 一句話概念

> 玩家操控第三人稱角色，用一個消耗魔力、有冷卻的火球技能攻擊會受擊反饋的訓練假人，體驗「技能戰鬥」的節奏。

這是一個**戰鬥手感的實驗場（lab）**，聚焦在「一個技能做到位」，而不是堆功能。

---

## 2. 這個原型想證明什麼

| 想證明的能力 | 怎麼證明 |
|---|---|
| 真的動手做出可玩的東西 | 一個能 PIE 試玩 30 秒的戰鬥迴圈 |
| 對 UE5 GAS 有概念 | 火球以 GameplayAbility / Attribute / Effect 正規實作，非土法計時器 |
| 懂動作戰鬥的「手感」 | 受擊反饋、魔力/冷卻節奏、投射物命中感 |
| 有設計思考、懂取捨 | 明確的 Non-Goals；彈反機制以文件規劃（見 §7） |
| 工程底子可遷移 | 資料形狀、狀態管理、驗收標準以規格（spec）方式定義 |

---

## 3. 核心玩法迴圈（Core Loop）

```
        ┌─────────────────────────────────────────┐
        │                                          │
   移動/瞄準 ──▶ 按左鍵放火球 ──▶ 消耗魔力 ──▶ 進入冷卻
        ▲                          │
        │                          ▼
        └──── 打中假人 ◀── 火球飛行命中 ── 假人受擊/扣血
                 │
                 ▼
         假人血歸零 → 短暫失能 → 原位重置 → 可再次試玩
```

**節奏設計意圖**：魔力上限與冷卻讓玩家「不能無腦連發」，必須抓放技能的時機——這是動作戰鬥「節奏感」的最小體現。

---

## 4. 系統設計

### 4.1 火球技能（GAS）

| 元素 | 實作 | 說明 |
|---|---|---|
| 觸發 | GameplayAbility，綁滑鼠左鍵 | 施放前檢查魔力與冷卻 |
| 投射物 | ProjectileMovement + 碰撞 | 朝角色面向飛行，命中即銷毀 |
| 魔力消耗 | Cost GameplayEffect | 扣固定 Mana，不足則不觸發 |
| 冷卻 | Cooldown GameplayEffect | 冷卻中再按無效、不扣魔力 |
| 傷害 | 命中時對假人套用 Damage Effect | 扣假人 Health |

**為什麼用投射物而非即時射線**：投射物有飛行時間，製造「瞄準 → 飛行 → 命中」的空間感與命中反饋，手感較好；也為未來彈反（假人發射投射物、玩家彈回）共用同一套投射物基礎。

### 4.2 訓練假人

- **生命值**：GAS Health / MaxHealth，受擊下降、下限為 0。
- **受擊反饋**：受擊當下播放可見反饋（閃白或擊退，擇一），讓玩家明確感到「打中了」。
- **死亡重置**：Health 歸零 → 短暫失能 → 原位重置滿血，使 demo 可重複試玩。
- **明確不做**：假人沒有 AI、不會移動、不會攻擊（本原型只驗證玩家端戰鬥反饋）。

### 4.3 戰鬥 HUD（UMG）

- 玩家**魔力條**：反映 Mana / MaxMana，施放後即時下降。
- 假人**血條**：反映 Health / MaxHealth，受擊下降、重置回滿。
- 原則：HUD 顯示值與 GAS Attribute 實際值必須一致，不脫節。

### 4.4 資料形狀（給工程視角的讀者）

```
AttributeSet (C++)
  Health / MaxHealth      // 假人
  Mana   / MaxMana        // 玩家

Fireball GameplayAbility (BP)
  ├─ CostEffect      → 扣 Mana
  ├─ CooldownEffect  → 冷卻計時
  └─ 命中 → DamageEffect → 扣目標 Health
```

## 5. 設計決策（為什麼這樣做）

這一節是本文件的重點——它展示的是設計判斷，而非功能清單。

### 5.1 為什麼用 GAS，而不是純 Blueprint 計時器
GAS 是連線技能戰鬥的官方標準，也是目標型遊戲（連線 + 3D + 技能）的核心技術棧。即使本 demo 不連線，用 GAS 實作火球能以「正規做法」呈現魔力/冷卻（Attribute + GameplayEffect），直接對上此類職位對 GAS 概念的要求。
**取捨**：純 Blueprint 更快，但無法展示 GAS 概念，故不採用。（若硬體/時間受限，保留純 BP 版火球為降級退路，確保「至少有可玩 demo」。）

### 5.2 為什麼只做「一個技能」
動作戰鬥的品質不在技能數量，而在單一技能的手感是否到位。與其做三個半吊子技能，不如把一個火球的施放、消耗、冷卻、命中反饋做完整——這更能證明對「戰鬥感」的理解。

### 5.3 為什麼假人不會還手（但仍要有反饋）
加入敵人 AI 會讓範圍爆炸（尋路、攻擊決策、平衡）。本原型聚焦「玩家端的攻擊手感」，因此假人只被動受擊，但**保留受擊反饋與血條**——因為「打得爽不爽」正是要驗證的東西。

### 5.4 為什麼彈反只寫文件、不實作（本次）
彈反是動作戰鬥中最能展現設計理解的機制，但完整實作（時機窗、判定、彈回）會超出原型範圍。策略是：用文件把彈反設計講清楚（§7），用 demo 證明動手能力——**用文件展示視野，用原型證明執行**，投報率最高。

## 6. 技術範圍與 Non-Goals

**技術棧**：Unreal Engine 5.x · Gameplay Ability System · Blueprint（主）+ 少量 C++（AttributeSet）· UMG · 第三人稱範本為基礎。

**開發環境**：於 Apple Silicon Mac（M1 / 16GB）開發；關閉 Lumen/Nanite、維持精簡場景以確保流暢。UE5 專案跨平台，可於 Windows 開啟。

**明確不做（守住範圍）**：

- ❌ 連線 / 多人
- ❌ 程序化關卡、Roguelite 道具/隨機
- ❌ 敵人 AI、尋路、攻擊
- ❌ 存檔、主選單、結算畫面
- ❌ 自製美術（使用引擎內建素材）
- ❌ 彈反機制的實作（僅文件規劃，見 §7）

## 7. 下一步：彈反機制（Roadmap，本次不實作）

> 呼應目標公司作品《靈能哨衛：無限》的核心——「弓變鐮刀、彈反敵人遠程攻擊」。此節展示若延續開發，會如何設計此機制。

**概念**：訓練假人升級為「會發射慢速投射物」，玩家在投射物接近時按下彈反鍵，於**時機窗口**內成功則將其彈回、命中假人造成傷害；失敗則自己受擊。

**設計要點**：

| 面向 | 設計 |
|---|---|
| 時機窗口 | 投射物進入角色前方一定距離時開啟約 0.2–0.3 秒的判定窗；參考《靈能哨衛》「判定寬鬆以照顧各程度玩家」的可及性理念 |
| 成功反饋 | 命中特寫/頓幀 + 投射物高速彈回，強化「擋下的爽感」 |
| 失敗反饋 | 玩家受擊、扣血，形成風險/回報 |
| 與現有系統的關係 | 直接複用火球的**投射物基礎**與 GAS **Attribute/Effect**——彈反是一個新的 GameplayAbility，判定成功時對投射物套用「反向速度 + 改變 instigator」 |

**為什麼這個機制值得做**：它把戰鬥從「單向輸出」變成「攻防節奏」，是動作 Roguelite 手感的關鍵；也證明開發者能拆解並致敬一款現有遊戲的核心設計。
