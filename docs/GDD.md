# Parry Lab — Game Design Document

> 一個用 Unreal Engine 5 + Gameplay Ability System（GAS）打造的微型戰鬥原型。
> 聚焦在「一個技能做到位」的動作戰鬥手感，而非完整遊戲。

---

## 1. 一句話概念

> 玩家操控第三人稱角色，用消耗魔力、有冷卻的火球技能攻擊會受擊、會重生的訓練假人，體驗「技能戰鬥」的節奏。

這是一個**戰鬥手感的實驗場（lab）**：把火球一個技能的施放、消耗、冷卻、命中、回饋做完整，而不是堆功能數量。

---

## 2. 核心玩法迴圈（Core Loop）

```
   移動/面向 ──▶ 左鍵連發火球 ──▶ 消耗魔力 ──▶ 進入冷卻
       ▲                              │
       │                              ▼
       └──── 打中假人 ◀── 火球飛行命中 ── 假人扣血
                │
                ▼
        假人血歸零 → 短暫失能 → 原位重生 → 可再次試玩
                                （魔力隨時間回復，支撐持續輸出）
```

**節奏設計意圖**：魔力上限、回復速度與冷卻共同決定「能連發多久、何時得停」。這是動作戰鬥「節奏感」的最小體現。

---

## 3. 系統設計

### 3.1 火球技能（GAS）

| 元素 | 實作 | 說明 |
|---|---|---|
| 觸發 | GameplayAbility，綁滑鼠左鍵 | 施放前檢查魔力與冷卻 |
| 投射物 | 球形碰撞 + ProjectileMovement | 朝角色面向飛行，命中即銷毀 |
| 一次多發 | 單次施放生成多顆、左右散射 | 可調數量與散射角度 |
| 魔力消耗 | Cost GameplayEffect | 扣固定 Mana，不足則不觸發 |
| 冷卻 | 施放後掛冷卻標籤、限時移除 | 冷卻中再按無效、不扣魔力 |
| 傷害 | 命中時對目標套用 Damage Effect | 扣目標 Health |

### 3.2 訓練假人

- **生命值**：GAS Health / MaxHealth，受擊下降、下限為 0。
- **死亡重置**：Health 歸零 → 短暫隱藏失能 → 原位重置滿血，使原型可重複試玩。
- **自動生成**：開場由 GameMode 於出生點前方自動排出數個假人，免手動擺放。
- **不做**：假人沒有 AI、不會移動、不會攻擊（本原型只驗證玩家端的攻擊手感）。

### 3.3 玩家角色與資源

- **魔力（Mana）**：施放火球消耗，並隨時間自動回復，形成「輸出／等待」的節奏。
- **生命（Health）**：屬性已具備，供未來加入敵人反擊時使用。

### 3.4 攝影機

- **固定斜俯視（3/4 視角）**：類似動作 roguelite（如 Hades、Diablo）的鏡頭。
- **不隨滑鼠或角色轉動**：維持穩定角度，降低操作負擔與 3D 暈眩，並讓火球飛行清楚可見。

### 3.5 資料形狀（GAS）

```
AttributeSet (C++)
  Health / MaxHealth      // 假人（與玩家）
  Mana   / MaxMana        // 玩家

Fireball GameplayAbility
  ├─ Cost Effect     → 扣 Mana
  ├─ 冷卻標籤         → 限時阻擋再次施放
  └─ 命中 → Damage Effect → 扣目標 Health
```

## 4. 設計決策（為什麼這樣做）

### 4.1 用 GAS，而不是純 Blueprint 計時器
GAS 是連線技能戰鬥的官方標準框架。即使本原型為單機，用 GAS 實作火球能以「正規做法」呈現屬性、消耗、冷卻、傷害，並讓日後擴充（連線、更多技能）有一致的基礎。
**取捨**：純 Blueprint 計時器更快，但無法沿用到更大的系統，故不採用。

### 4.2 只做「一個技能」
動作戰鬥的品質不在技能數量，而在單一技能的手感是否到位。與其做三個半吊子技能，不如把火球的施放、消耗、冷卻、命中、回饋做完整。

### 4.3 假人不會還手，但仍要有回饋
加入敵人 AI 會讓範圍爆炸（尋路、攻擊決策、平衡）。本原型聚焦「玩家端的攻擊手感」，因此假人只被動受擊、受擊後重生——因為「打得爽不爽、看不看得出打中」正是要驗證的東西。

### 4.4 固定攝影機
可自由旋轉的第三人稱鏡頭雖然常見，但會增加操作負擔。固定斜俯視讓玩家專注在移動與瞄準、看清全場，也讓原型的核心（戰鬥）更容易被觀察與展示。

## 5. 下一步：彈反機制（Roadmap，尚未實作）

延續開發時，最想加入的是**彈反（parry）**——動作 roguelite 中最能展現設計理解的機制之一。

**概念**：訓練假人升級為會發射慢速投射物；玩家在投射物接近時按下彈反鍵，於**時機窗口**內成功則將其彈回、命中造成傷害；失敗則自己受擊。

| 面向 | 設計 |
|---|---|
| 時機窗口 | 投射物進入角色前方一定距離時開啟約 0.2–0.3 秒判定窗；判定寬鬆以照顧各程度玩家 |
| 成功回饋 | 命中特寫／頓幀 + 投射物高速彈回，強化「擋下的爽感」 |
| 失敗回饋 | 玩家受擊、扣血，形成風險／回報 |
| 與現有系統的關係 | 複用火球的**投射物基礎**與 GAS **Attribute/Effect**：彈反是新的 GameplayAbility，成功時對投射物套用「反向速度 + 改變 instigator」 |

**為什麼值得做**：它把戰鬥從「單向輸出」變成「攻防節奏」，是動作 roguelite 手感的關鍵。

## 6. 技術範圍與 Non-Goals

**技術棧**：Unreal Engine 5.8 · Gameplay Ability System · C++（屬性、技能、效果、假人）+ Blueprint · 第三人稱範本為基礎。

**明確不做（守住原型範圍）**：

- ❌ 連線 / 多人
- ❌ 程序化關卡、道具 / 隨機（roguelite 內容）
- ❌ 敵人 AI、尋路、攻擊
- ❌ 存檔、主選單、結算畫面
- ❌ 自製美術（使用引擎內建素材）
- ❌ 彈反機制的實作（僅文件規劃，見 §5）
- ⏳ 正式血條 / 魔力條 UI（目前以螢幕除錯文字顯示數值）
- ⏳ 假人受擊的視覺回饋（閃白 / 擊退）

## 7. 開發紀錄（踩過的坑與解法）

原型於 Apple Silicon Mac（M1 / 16GB）以 spec-driven 方式開發，過程中的主要挑戰與解法：

- **Xcode 26 缺 Metal 工具鏈**：UE 5.8 啟動時報 `missing Metal Toolchain`。解法：`xcodebuild -downloadComponent MetalToolchain` 下載按需元件（Xcode 16.3+ 將 Metal 編譯器改為獨立下載）。
- **GameplayEffect 在建構子加元件會 crash**：於 `UGameplayEffect` 建構子呼叫 `FindOrAddComponent<UTargetTagsGameplayEffectComponent>` 會因 CDO 階段 `NewObject` 空名而 fatal。解法：改用 **loose gameplay tag + 計時器**實作冷卻，並覆寫 `GetCooldownTags`，行為等價且不再 crash。
- **UE 視為錯誤的警告**：`for` 迴圈內立即 `break` 觸發 `-Wunreachable-code-loop-increment`。解法：改用單次取值（`TActorIterator` 直接取第一個）而非迴圈。
- **鏡頭與操作手感迭代**：初版自由旋轉第三人稱鏡頭造成暈眩與操作困難 → 改為固定斜俯視（3/4 視角），並把火球改為朝角色面向發射。
- **命中不明顯**：火球初版小、生成貼身、飛太快 → 放大、拉遠生成點、抬高，並用固定鏡頭讓飛行清楚可見。
